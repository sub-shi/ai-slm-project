1.2 Conviction Validity Analysis

Conviction is not a softmax probability. It is a floating-point scalar generated as text by the language model and parsed deterministically. Validity is assessed on three axes: (1) range validity — all values must fall in [0.0, 1.0]; (2) distribution shape — values concentrated at boundaries (0.0 or 1.0) indicate degenerate output; (3) monotonic reliability — directional signals accompanied by higher conviction should correspond to greater behavioral consistency when evaluated in aggregate. A reliability diagram is reported but is not the primary validity criterion; the calibration narrative later in this section addresses why the conviction value is a design artefact, not a probabilistic output.

2.2 Finding 1 — Conviction Field Type Inconsistency (Critical)

Semantic validation of output fields revealed that the conviction field was not consistently typed. A subset of samples encoded conviction as a JSON string rather than a numeric float. Since the orchestrator performs deterministic numeric thresholding (conviction < 0.40), any non-numeric conviction value arriving at inference time would cause a type error or silent misclassification.

Inspection of the removed samples list confirmed that rows 48–69 and additional scattered rows used qualitative conviction descriptors rather than numeric values: 'high', 'moderate', 'low', 'high confidence', 'moderate confidence', 'strong', 'weak'. These appeared in a contiguous block (rows 48–69) plus isolated later occurrences, strongly suggesting two distinct upstream annotation or generation pipelines were merged without schema normalisation.

Finding: 39 of 300 samples (13%) contained qualitative conviction labels incompatible with the production schema. The issue was localised to output-field semantics; input features were unaffected. The corruption appeared in a contiguous block, indicating a pipeline merge artefact rather than random annotation error.

Decision: Samples where conviction could not be unambiguously extracted as a float in [0, 1] were excluded from training. The cleaning pipeline applied a two-stage resolution: (1) attempt direct float() cast — succeeds for already-numeric values; (2) apply regex r"\s*(\d+\.?\d*)\s*(?:\(.+\))?\s*" to recover numeric values embedded in annotated strings such as "0.8 (high)". Purely qualitative labels with no embedded numeric were removed rather than heuristically remapped, since any mapping would introduce fabricated supervision signal. Final cleaned set: 261 samples (39 removed).

Alternative considered and rejected: Mapping qualitative labels to fixed numeric equivalents (e.g., 'high' → 0.8) was considered but rejected. The mapping would be arbitrary; it would introduce correlated conviction noise across all remapped samples; and the clustered nature of the corruption suggests a distinct labeling convention that may not correspond to the numeric scale used elsewhere in the dataset.

2.3 Finding 2 — Class Imbalance

Direction label distribution across 300 raw samples: NEUTRAL ≈ 43%, CE ≈ 31%, PE ≈ 26%. The NEUTRAL class dominates; PE is the smallest class. This may bias the model toward conservative predictions. However, the imbalance plausibly reflects real market conditions — non-directional periods are genuinely frequent in NIFTY options data — and no resampling was applied, as resampling would distort the empirical base rate the model is meant to learn.

2.4 Finding 3 — ADX Regime Coverage Gap (Critical for Orchestrator Design)

All inspected ADX values across 300 training samples fell in the range 23.66–30.39, entirely above the orchestrator's suppression threshold of 20. The model was therefore trained exclusively on trend-present regimes and has never seen a market state where ADX < 20. This is not a data quality issue per se but a distribution coverage gap with direct safety implications: the orchestrator's ADX suppression rule is not merely a filter — it is a hard boundary preventing model inference in regimes the model has never encountered during training. This finding informed the later safety analysis section.

3 · Fine-Tuning and RAG

3.1 Model Choice: TinyLlama-1.1B-Chat-v1.0

TinyLlama-1.1B was selected over Phi-2 for three reasons. First, the inference constraint requires CPU deployment with 4-bit quantisation: TinyLlama's 1.1B parameter count fits comfortably within this envelope while Phi-2's 2.7B count would produce meaningfully higher per-inference latency in a latency-sensitive trading context. Second, the task is output-format conditioning — the model must produce valid JSON of a fixed schema — rather than complex multi-step reasoning, favouring a smaller model with tighter output control. Third, TinyLlama-Chat was already instruction-tuned, providing a capable prompt-following base that reduces the supervision required from the relatively small cleaned training set of 261 examples.

3.4 The Conviction Field as a Design Problem

The conviction value is the field most likely to be misunderstood in this system. It is generated by the language model as a sequence of text tokens — the characters 0, ., 4, 7 — and parsed back to a float by the orchestrator. It is therefore meaningfully different from a softmax probability in two ways.

First, softmax over the direction token would give the model's token-level uncertainty about whether the next character is C, P, or N — a measure of lexical ambiguity given the preceding prompt, not a measure of market-regime uncertainty. It does not capture whether the model's internal representations are far from anything seen in training, nor whether the market features are conflicting or consistent. Second, in a fine-tuned autoregressive model, the conviction token is produced after the direction token has already been generated and is therefore conditioned on the model's own prior output — it is more accurately described as a self-reported confidence annotation than an independent probability estimate.

To make conviction informative, the training data was cleaned so that all conviction values are valid floats in [0.0, 1.0], preserving the empirical distribution from the original dataset (clustered 0.4–0.6). The model learns to associate market feature configurations with conviction levels from the training examples. Validation checks that the generated conviction is parseable and in-range; the orchestrator's downgrade rule at 0.40 provides a hard floor below which the direction signal is suppressed regardless of model output.

The honest limitation is stated here: conviction is not calibrated in the frequentist sense. It is a learned annotation that may correlate with the model's internal uncertainty but is not guaranteed to do so. Section 5 discusses what would be required to make it genuinely calibrated.

3.6 RAG Experiment

The retrieval layer provided in the assignment bundle was used without modification via retrieve(market_state, k=3), consistent with assignment constraints. Retrieved episodes were drawn from rag_corpus.jsonl and incorporated into the prompt as contextual summaries rather than direct target-output JSON.

The retrieval prompt included:

  • market regime labels
  • historical market-state summaries
  • retrieved outcome descriptions
  • current market state
  • explicit schema constraints

An important design decision was intentionally avoiding direct inclusion of historical output JSON. Early prompt versions caused the compact SLM to copy retrieved target outputs nearly verbatim, including timestamps and signal identifiers. Replacing retrieved target JSON with descriptive outcome summaries reduced direct copying behaviour.

However, retrieval augmentation introduced a second failure mode: schema instability. Under retrieval conditioning, TinyLlama occasionally:

  • repeated schema keys multiple times
  • continued generation beyond the required JSON object
  • hallucinated additional fields such as context and return
  • truncated generation before closing braces

To stabilize inference, deterministic post-processing was added:

  • generation truncation at the first closing brace
  • higher max_new_tokens allocation for long retrieval prompts
  • explicit schema examples using concrete values rather than symbolic placeholders

The final RAG experiment produced an important systems insight: retrieval augmentation increased behavioural diversity and occasionally disrupted the model's pure NEUTRAL collapse, but also weakened strict schema discipline in the compact SLM.

This highlighted a meaningful tradeoff in lightweight financial RAG systems: retrieval context can improve contextual grounding while simultaneously increasing structural generation entropy.

5 · Safety Analysis

5.1 Scenario: 09:30 Expiry Thursday, VIX +3σ, ADX = 14

Walk through the system from market state ingestion to final orchestrator output.

Stage 1 — Market State Ingestion

The orchestrator receives a market state dictionary with the nine required fields. In this scenario, the key values are: adx_14 = 14 (below suppression threshold), vix_india at approximately 3 standard deviations above the 30-day trailing mean (approximately 19–21 given the training dataset range), dte_nearest = 0 (expiry day), and elevated atm_iv and realized_vol_5d consistent with a VIX spike. The state is passed to the orchestrator function as a Python dictionary alongside a formatted prompt string.

Stage 2 — Rule 1: ADX Suppression Check

The orchestrator evaluates market_state["adx_14"] < 20. With ADX = 14, this condition is True. The orchestrator immediately calls log_event("LOW_ADX_SUPPRESSION", {"adx": 14}) and returns neutral_response("LOW_ADX_SUPPRESSION"):

{"direction": "NEUTRAL", "conviction": 0.0, "horizon": "intraday",
 "signal_id": "<uuid4>", "generated_at": "<ISO timestamp>",
 "reason": "LOW_ADX_SUPPRESSION"}

The model is never called. The downstream pipeline receives NEUTRAL with conviction 0.0. The log entry records the ADX value that triggered suppression and the timestamp. This is the correct and complete system response for this scenario under the current implementation.

Stage 3 — What Does Not Happen (and Why That Matters)

The model inference stage (Rule 2 parse check, Rule 3 conviction downgrade) is never reached. This means the elevated VIX condition — the most structurally significant feature of this scenario — has no influence on the output path. The suppression is triggered by ADX alone. This is correct behavior given the orchestrator specification, but it conceals an important limitation: the system cannot distinguish between (a) a low-ADX day with normal VIX, which is a routine non-directional period, and (b) a low-ADX expiry-day VIX spike, which is a genuine tail-risk event. Both produce identical orchestrator output.

5.2 What Is Wrong With the Current Implementation

Problem 1 — Neutrality Collapse

The fine-tuned model collapses to NEUTRAL across all evaluated samples. In the expiry-Thursday VIX-spike scenario, this would not matter because the orchestrator suppresses before model inference. But in a high-ADX VIX-spike scenario (ADX > 20, VIX elevated), the model would be called and would produce NEUTRAL with a conviction near the 0.40 threshold, which the orchestrator would downgrade to NEUTRAL. The system would fail silently — producing NEUTRAL not because the orchestrator correctly identified a dangerous regime, but because the model has learned a near-constant output. The log would record LOW_CONVICTION_DOWNGRADE rather than any VIX-conditioned suppression, making post-hoc diagnosis misleading.

Problem 2 — VIX Spike Is an Unseen Distribution

The training dataset VIX range is 12.67–15.37. A VIX reading 3σ above the 30-day mean on an expiry day would likely exceed this range. The model has never seen a market state with this VIX level during fine-tuning. The orchestrator's ADX suppression catches this specific scenario (ADX = 14), but if ADX were 22 and VIX were spiking, the model would be called in an out-of-distribution state with no mechanism to flag the distributional shift. The conviction field would not reliably reflect this uncertainty because it was learned from in-distribution examples.

Problem 3 — Expiry Day (DTE = 0) Is Underrepresented

Expiry Thursdays are structurally distinct — gamma risk is elevated, options pricing mechanics differ, and intraday moves are amplified. The training data contains DTE values across a range, but the expiry-day sub-distribution may be sparse. The model has no explicit regime indicator for expiry conditions.

Problem 4 — Orchestrator Cannot Distinguish Tail-Risk Events

The three orchestrator rules are unconditional: ADX < 20 always suppresses; parse failure always returns NEUTRAL; conviction < 0.40 always downgrades. There is no rule that suppresses the model based on VIX level, on IV spike magnitude, or on DTE proximity to zero. A VIX reading 3σ above mean on expiry Thursday is arguably more dangerous than a low-ADX non-directional day, yet the current orchestrator would handle the former by calling the model (if ADX > 20) and the latter by suppressing it. The logic is inverted for tail-risk events.

5.3 Suppression Rate Analysis

Observed ADX suppression rate in the 50-sample evaluation: 0%. This is expected given that the training data contained no ADX values below 20, and the evaluation samples were drawn from the same distribution. The suppression rule therefore had no opportunity to fire. This does not mean the rule is non-functional — it is correctly implemented and would suppress at ADX < 20 — but the evaluation dataset provides no evidence of whether the suppression boundary is correctly calibrated for real market conditions that include low-trend periods.

Low-conviction downgrade rate: ~19% (50/261). This is within the pre-committed threshold and reflects the model's tendency to generate conviction values near the 0.40 boundary. Elevated downgrade rates would be expected in high-VIX regimes where the model's learned feature-to-conviction mapping may be less reliable — but this regime was not represented in the evaluation set.

5.4 What I Would Build Next

To handle this class of event correctly, four additions are required. First, a VIX regime suppression rule in the orchestrator: if VIX exceeds a configurable threshold (e.g., trailing 30-day mean + 2σ), suppress regardless of model output and log HIGH_VIX_SUPPRESSION. Second, a DTE proximity rule: suppress or flag signals when DTE = 0 or 1 (expiry and pre-expiry), since the model has insufficient training coverage of these conditions. Third, a distributional shift detector: compute feature-space distance between the inference-time market state and the nearest training example; if distance exceeds a threshold, suppress with OOD_SUPPRESSION and log the distance. Fourth, conviction recalibration: train a lightweight isotonic regression or temperature-scaling layer on a held-out calibration set to map raw model conviction outputs to empirical accuracy rates, making conviction a genuinely calibrated probability rather than a learned annotation.

The current system is safe within the bounds of its training distribution and benefits from correct orchestrator logic. It is not yet safe for tail-risk events it has not seen. This boundary is known, logged, and documented. That honesty is itself a safety property.
